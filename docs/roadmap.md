# Roadmap — From Starter Kit to Living Engine

This roadmap turns the [high-level design](./hld.md) into an ordered ladder. Each rung is **one fully-functional idea**, published as a tag plus a GitHub Release, before the ladder climbs to the next. No half-shipped surfaces spread across versions.

**Where we are:** mid-Q3 2026, at **v2.17.0**. **Loop 1 is complete** — v2.16 shipped the confirm → apply → verifier-gate → rollback machinery — and the **Steward shipped partially at v2.17** (auto-cleaning delivered; living-organization and promote-repetitive-to-Skill deferred to a later rung by an owner scope decision at the gate). The near-term rungs (**v2.18 → v2.19**) are realistic for this quarter at the kit's cadence, and **pull-updates (v2.19) now lands in Q3** because it is on the critical path to the engine. **Community intake (v2.20) is demand-gated** — it ships when real contributors ask for it, not on a fixed quarter slot. **v3.0 — the spawn-only engine — is the Q4 headline**, the release where "starter kit" becomes "engine." The **hub view is decoupled into a later rung (v3.x)**: a **completed feasibility spike** found no native cross-project capability (Cowork Projects are hard-isolated), so v3.x is re-scoped as a **local-filesystem hub** — design settled, timing open.

**How to read the classification column.** Every rung that writes an instruction file, ingests external content, or spawns a space is **SECURITY-SENSITIVE** — it carries a mandatory, permanent security review and a required audit, from its first line, per the design principles in the HLD. That is standing, not per-cycle.

## Parallel track (complete)

- **Hub-feasibility doc-spike (PURE-DOC) — COMPLETE.** Ran *alongside* the ladder to answer **KDQ-HUB**: *can* a parent space see isolated child spaces at all? **Verdict: not natively — ever.** Cowork Projects are hard-isolated by platform design, with no cross-project reference mechanism and none announced. The workaround that survives is **local-only**, because Projects are backed by real local folders: (1) a hub space reads every sibling's on-disk files directly, and (2) each sibling writes a compact status card the hub reconstructs a view from (the `registry.json` + `pipeline.md` pattern The-Council uses on itself). This retired the program's biggest structural unknown on paper and re-scoped v3.x from "feasibility-gated" to "local-filesystem design, timing open." *(Sources, trust-model level: the Claude Cowork / Projects support docs on isolation, Anthropic's Projects announcement, and the Claude Code cross-project reference request, issue #68262.)*

## Rung notes

- **v2.16 (shipped 2026-07-20, Loop 1 Increment 2).** The security posture *inverted* at this rung: v2.15's safety was structural (the loop could only write its own ledger), and v2.16 deliberately opened a write channel to instruction files. It was the biggest single blast-radius escalation in the kit's history *so far*, and the whole cycle's weight landed on the Phase-2 gate. Everything above it reuses the confirm → apply → verify → rollback machinery it established — it was built well, so the rest of the ladder inherits it.
- **v2.17 (Steward)** depends on v2.16, not on the substrate — its writes are confirmed local file moves plus the already-gated promote-to-Skill path, so it can ship before the contribution substrate exists. It is the fastest way to turn v2.16's machinery into visible single-user value.
- **v2.18 (Substrate)** is the pivot, and it is deliberately **slim**: canonicalization + deterministic scan + the lock-trichotomy + the per-workspace install manifest + a *minimal* tier schema. The **LLM-judge is left out on purpose** — on curated-only traffic the maintainer's own review is strictly stronger, so the judge would add cost without safety until untrusted content actually arrives at intake (v2.20). Keep the community-tier schema minimal too — just enough for curated to ride — so nothing speculative is built ahead of v3.1.
- **v2.18 transferability constraint (binding input to its `/spec`).** The substrate must be designed so an *external* consumer can pull skills and operating knowledge from it — not just Cowork's own wizard. The first such consumer is **Confidante**, a local sovereign shell running a ~9B model (`qwen3.5:9b`), not Claude. Two sub-constraints: **(a)** author skills against the **open SKILL.md standard**, never Cowork-private extensions — any Cowork-specific field (wizard hooks, preset routing) lives in a *surface adapter*, not in the skill body a foreign runtime reads; **(b)** the registry/manifest (`tier` / `source_url` / `sha256`, the KDQ-MANIFEST deliverable) must be consumable by an **external puller**, not only the wizard. Name the risk so it is not treated as happy-path: **format transfer** (SKILL.md) is free, but **capability transfer** — does the skill actually *work* on the target runtime? — is **not** bought by format; it needs a **target-runtime eval gate** (reuse the v2.13 WS-EVAL grade-before-install muscle). "It's in SKILL.md format" ≠ "it runs on Confidante." Carried as **KDQ-XFER** (HLD §12, §4).
- **v2.19 (Pull)** ships **before** intake, on purpose (see Sequencing). It is the first consumer of the v2.18 install manifest, which is exactly why building the manifest at v2.18 with a waiting consumer keeps it honest. Pull is safe by construction — it distributes only curated content — and it is what makes "spawn a new space with the latest skills" real by the time v3.0 arrives.
- **v2.20 (Intake)** is **demand-gated, not calendar-gated**: it ships when roughly five organic "how do I contribute?" issues arrive from real users, not on a fixed quarter. It is where the LLM-judge is introduced, because it is the first rung where genuinely untrusted content flows in. Until then the curated pool still grows — via maintainer authoring and the shipped `PROMOTE.md` ceremony.
- **v3.0 (Engine, spawn-only)** needs its own dedicated design cycle — its own `/spec` and `/plan`. It is the least-designed rung, and the ladder is built so nothing below it depends on its internals. The one open question that stays with it is **KDQ-SPAWN-SEC** (a spawn gate stronger than Loop 1's apply). The hub view is *not* here — it is decoupled to v3.x so a platform-isolation question (which the spike has since settled: no native path exists) could never hold the North-Star release hostage.
- **v3.x (Hub view)** is no longer blocked on a feasibility unknown — the spike settled **KDQ-HUB**: a native cross-project view is impossible, so v3.x is a **local-filesystem hub** (read sibling folders + status cards). Its design is settled; only its scheduling is open. This is why v3.0 spawn carries the shared-parent-directory + status-card obligation — it is the cheap groundwork that keeps v3.x from needing every space re-spawned.

## Sequencing — RESOLVED

Two orderings were settled:

1. **Shared-substrate-first (holds).** Build the trust machine once, then layer the two directions on it — rather than build a contribution pipeline and an update pipeline separately.
2. **Pull before intake (KDQ-SEQ — RESOLVED, pull-first).** After the substrate, **pull-updates (v2.19) ships before community intake (v2.20).** The reasons:
   - **Critical path.** Pull is on the path to the v3.0 engine — spawning "with the latest skills" chains pull first. Intake, by contrast, feeds only the *later* community tier (v3.1). Pull unblocks the headline release; intake does not.
   - **Forces the manifest closed early, with a real consumer.** Pull is the first thing that actually needs the per-workspace install manifest (KDQ-MANIFEST), so shipping pull first means the manifest is designed against a live consumer instead of in the abstract.
   - **The cheaper security spend.** Pull distributes only curated content, so it is safe by construction; it does not need the LLM-judge or the untrusted-content machinery that intake requires.
   - **The empowerment yardstick.** Pull reaches *every existing user* (their spaces stay alive and updated); intake reaches only the smaller set of people who want to contribute.

Intake is then demand-gated on top of that ordering: even though it comes after pull in the ladder, it only *ships* when real contribution demand appears.

## Later (not yet versioned)

- **Hosted guided web form** — a better non-coder contribution experience than the issue-form, taken only if submission volume justifies the real build cost. An upgrade to v2.20's intake, not a replacement for it.
- **Sigstore / provenance identity binding** — proves *who* signed content and makes it revocable. It proves origin, not intent, so it adds no day-one safety against injection; it earns its place once the community tier is open and identity/revocation matter.
- **Confirmation batching (KDQ-BATCH)** — a fatigue-reducer for users facing many individual confirmations. The *constraint* (confirm every self-modification individually) is in force from v2.16; the batching *feature* stays deferred until the confirmation surface's integrity over time is worked out.
- **OSS / self-hostable, sovereign-user alternative** — a deferred, long-roadmap option: an open-source, self-hostable Cowork engine a sovereign user can run on their own terms. It is deferred *by design*, not by neglect: the ladder is **depth-first** on purpose, because the guided setup + curation + trust layer is the part that does *not* commoditize, and building it deep first is what gives an eventual OSS release a real moat rather than a thin lowest-common-denominator wrapper. It fits the sovereign-domain thesis — a self-hostable, tool-agnostic engine is the natural constituent for a sovereign-user picture — so the OSS move is the kit *earning* its place there, not a detour. **Two separable dials, which must not fuse:** **(A) OSS / self-host** — run the same Claude-native engine on your own terms; and **(B) cross-tool portability** — run it against runtimes other than Claude. The sovereign-user driver points hardest at **dial A**; **dial B is optional on top**. Do not couple them in the roadmap. The mechanical seam for dial B already exists dormant: the `improvement-plan-2026-07-18.md` **Phase D** ("upstream refresh → multi-tool," consuming upstream converter/`tools.json` via a lock bump) is the lever if and when dial B is picked. The v2.18 transferability constraint (above) is the near-term down-payment on this: authoring to the open SKILL.md standard is what keeps dial A coherent when it is eventually taken.

## Standing principle (not a version)

The kit continuously tracks Claude Cowork's own product evolution and community best practices and folds them in. That is the ongoing cost of being the reference framework — it is a permanent posture, not a rung on the ladder, and it runs alongside every version above.
